Product Requirements Document
Defines the vision, goals, scope, and requirements for Trackly MVP — targeting a 1-month build cycle.
💡
Mission statement
Trackly helps freelancers and small teams effortlessly track time, understand where hours go, and get paid accurately — without the complexity of enterprise tools. Ship in 1 month, delight on day one.

Problem statement
Freelancers and small teams lose money because time tracking tools are either too complex (Jira), too expensive (Harvest), or too bare-bones (spreadsheets). No existing tool is simultaneously simple, affordable, and insightful.

Tickspot is easy to use but lacks modern UX, reporting, and integrations with project management tools (Trello, Asana, Jira etc.). Toggl and Harvest are feature-rich but overwhelming for non-enterprise users. This leads to inconsistent tracking, forgotten billable hours, and lost revenue.
⚠️
Key pain points
Forgetting to log time · No visibility on billable vs non-billable hours · Projects run over budget without warning · Manual invoice creation from disconnected time data · Clunky UX that discourages consistent tracking

Functional requirements
▶ ⏱ Time tracking core
One-click start/stop timer. Timer state persists across page refreshes via server-side state (DB) and local localStorage fallback. Manual entry supports custom date, start/end time or duration, and project assignment. Entries can be edited or deleted within 30 days. Each entry tagged as billable or non-billable. Minimum 1-minute duration to save. One active timer per user at a time.
▶ 📁 Projects & task management
Users create projects with name, client name, optional hourly rate, and total hour budget. Projects have a status: active, paused, archived. Budget alerts email-notify admin at 80% and 100% consumed. Project color picker for visual identification in reports.
▶ 👥 Team & workspace
Each user belongs to one workspace in MVP. Admin can invite members via email. Roles: admin (full access) and member (log own time only). Admin can set per-member hourly rates. Reports filtered by member visible to admin only. Pending invites expire after 7 days.
▶ 📊 Reports & exports
Summary dashboard: total hours, billable hours, billable revenue. Filter by date range, project, and team member. Bar chart breakdown by project using recharts. Export to CSV (free tier). Weekly Monday email digest for admin. Date ranges: this week, last week, this month, custom.
▶ 🔐 Auth & onboarding
Email/password signup via better-auth. Google OAuth SSO. Confirmation email on signup. Password reset flow. First-run onboarding wizard: create workspace → invite team (optional) → create first project → log first entry. New Google users auto-create workspace; existing users matched by email.
Non-functional requirements
⚡ Performance

📡 Availability
99.5% uptime SLA · Deploy on AWS · Planned maintenance: Mon 02:00–04:00 UTC
🔒 Security

♿ Accessibility
WCAG 2.1 AA · shadcn/ui components (Radix primitives) · Keyboard navigable
📈 Scalability
Support 10,000 workspaces · PostgreSQL connection pooling via AWS (Superbase) · Serverless-ready
🧪 Testing
Unit tests for business logic (Vitest) · E2E critical paths (Playwright) · CI via GitHub Actions

Core user stories
Epic 1 — Authentication
US-001P0
Sign up with email
As a new user, I want to create an account with my email so that I can start tracking time.
  • Register with email + password (min 8 chars)
  • Confirmation email sent via Resend on signup
  • Invalid email shows inline Zod validation error
  • Unverified users can log in but see a banner
US-002P0
Sign in with Google
As a returning user, I want to log in with Google so that I don't have to remember a password.
  • Google OAuth button visible on login screen
  • New Google users auto-create a workspace
  • Existing users matched by email, merged into account
  • Redirect to dashboard on success
Epic 2 — Time tracking
US-010P0
Start a timer
As a user, I want to start a timer with one click so that I can begin tracking without interrupting my flow.
  • Timer starts on click, shows elapsed time in header
  • User can optionally set project before starting
  • Timer state persists to DB on start (not just localStorage)
  • Only one active timer allowed per user at a time
US-011P0
Stop and save a timer
As a user, I want to stop my timer and save the entry so that the time is logged against the right project.
  • Stop button saves entry with duration, project, timestamp
  • User prompted to add notes and confirm billable flag in modal
  • Entry appears immediately in time log via optimistic update
  • Minimum 1-minute duration enforced
US-012P1
Log time manually
As a user, I want to log time for past work so I can capture hours I forgot to track in real time.
  • Form accepts date, start time, end time or duration
  • Can backdate up to 90 days
  • Project and notes fields available
  • Entry visible in time log immediately after saving
Epic 3 — Projects
US-020P0
Create a project
As an admin, I want to create a project with a budget so that I can track time spent vs planned.
  • Required fields: project name, client name
  • Optional: hourly rate, total hour budget, color label
  • Project visible to all workspace members immediately
  • Budget alerts set at 80% and 100% by default (email)
Epic 4 — Reports
US-030P1
View weekly report
As a user, I want to see a summary of my time this week so that I know if I'm on track with my workload.
  • Shows total hours, billable hours, and revenue for selected period
  • Breakdown by project as a bar chart (recharts)
  • Date range filter: this week, last week, this month, custom
  • Export to CSV on all plans; PDF on Pro

📌
Pattern decision: Server Actions vs Route Handlers
Server Actions are used for all mutations (create, update, delete) — simpler, no fetch needed. Route Handlers (app/api/**) are used only for data fetching that needs a cacheable URL (reports export, third-party webhooks, public API endpoints).

Auth Design
Authentication architecture using better-auth with email and Google OAuth.
🔐
Why better-auth over NextAuth?
better-auth provides a cleaner TypeScript API, first-class Prisma adapter, built-in session management with rotation, and no adapter inconsistencies. Config lives in lib/auth.ts and is shared between server and client.
Auth configuration
📦 Package
better-auth with Prisma adapter
🔑 Providers
Email/password · Google OAuth 2.0
🍪 Sessions
HTTP-only cookies · 30-day expiry · Rotation on use
🛡 Middleware
middleware.ts — protect all (dashboard) routes · Redirect to /login if unauthenticated
📧 Emails
Verification on signup · Password reset · Invite to workspace — all via Resend
🔒 Password
Bcrypt hashed · Min 8 chars enforced · No max (bcrypt handles)
Route protection
Route pattern Access Redirect on fail
/login, /signup Public — redirect away if authed /dashboard
/dashboard/** Auth required /login
/projects/** Auth + workspace membership /login
/settings/** Auth + admin role for workspace settings /dashboard
/api/auth/** Public (better-auth handles) —
/api/** Auth required — session checked via better-auth 401 JSON
